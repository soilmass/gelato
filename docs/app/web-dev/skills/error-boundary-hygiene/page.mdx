## When to Use

Activate when any of the following is true:
- Adding `error.tsx`, `global-error.tsx`, or `not-found.tsx` to a route segment
- Implementing a custom `class ErrorBoundary extends Component` wrapper
- A PR modifies an existing error file
- "My error page shows the default Next.js error instead of my component"
- "My reset button doesn't reset the route segment"

## When NOT to Use

Do NOT activate for:
- **Runtime error reporting** — `sentry-setup`
- **Try/catch discipline inside route handlers** — case-by-case judgment; out of scope
- **Loading / Suspense boundary design** — `suspense-streaming-boundaries`
- **Client-side fetch error UI** — belongs inside the component's own state machine, not an error boundary

## Procedure

### Step 1 — `error.tsx` / `global-error.tsx` must declare `'use client'`

Next.js requires error files to be Client Components — they run inside a React error boundary, which is a client-only construct. Missing the directive is a runtime 500.

```tsx
// app/dashboard/error.tsx — RIGHT
'use client';
export default function Error({ error, reset }: { error: Error; reset: () => void }) { … }

// WRONG — defaults to Server Component; Next.js errors at build time or runtime
export default function Error({ error, reset }) { … }
```

### Step 2 — `error.tsx` default export must accept `reset: () => void`

The `reset` prop is how Next.js hands the caller a way to re-render the failed segment. Without accepting it, a user clicking "Try again" has no way to recover. The signature must either destructure `reset` or accept the full props object (`\{ error, reset \}`).

```tsx
// RIGHT
export default function Error({ error, reset }: { error: Error; reset: () => void }) {
  return <button onClick={() => reset()}>Try again</button>;
}

// WRONG — reset absent
export default function Error({ error }: { error: Error }) {
  return <span>Oops</span>;
}
```

### Step 3 — `global-error.tsx` must wrap its JSX in `<html>` + `<body>`

Global error boundaries replace the root layout entirely. Without `<html>` and `<body>`, the resulting document is invalid and the browser falls back to its own error UI.

```tsx
// app/global-error.tsx — RIGHT
'use client';
export default function GlobalError({ reset }: { reset: () => void }) {
  return (
    <html lang="en">
      <body>
        <h1>Something went wrong</h1>
        <button onClick={() => reset()}>Try again</button>
      </body>
    </html>
  );
}

// WRONG — no html/body wrapper
'use client';
export default function GlobalError() {
  return <h1>Something went wrong</h1>;
}
```

### Step 4 — Custom class `ErrorBoundary` must implement a catch method

A React class boundary that extends `Component` must declare at least one of `componentDidCatch` (side effect) or `getDerivedStateFromError` (state derivation). A class without either is not actually a boundary — render errors pass through.

```tsx
// RIGHT — derived state + (optional) side effect
export class ErrorBoundary extends Component {
  state = { hasError: false };
  static getDerivedStateFromError() { return { hasError: true }; }
  componentDidCatch(err: Error) { reportToSentry(err); }
  render() { return this.state.hasError ? <Fallback /> : this.props.children; }
}

// WRONG — no catch method; behaves like a plain component
export class ErrorBoundary extends Component {
  state = { hasError: false };
  render() { return this.state.hasError ? <Fallback /> : this.props.children; }
}
```

## Examples

### Example 1 — `error-tsx-missing-use-client`

**Input:** `app/blog/error.tsx` without `'use client'` as its first directive.
**Output:** Missing directive means Next.js treats it as a Server Component — error-boundary behaviour doesn't attach. Fix: add `'use client';` as the first statement.

### Example 2 — `global-error-missing-html-body`

**Input:** `app/global-error.tsx` returns `<div>Error</div>`.
**Output:** Global error replaces the root layout including `<html>` and `<body>`. Fix: wrap in `<html lang="…"><body>…</body></html>`.

## Edge Cases

- **Custom wrapper around the Error component:** `export default function BlogError(props) \{ return <ErrorUI \{...props\} />; \}` — the classifier checks the default export signature; destructuring inside the wrapper function counts.
- **TypeScript prop interface:** `interface Props \{ error: Error; reset: () => void \}; export default function Error(\{ error, reset \}: Props)` is accepted.
- **Class-based error.tsx (non-idiomatic but legal):** checked the same — `reset` must be in the component's props signature.
- **`not-found.tsx`:** no `'use client'` requirement — it's a Server Component by default. The skill doesn't flag not-found files.
- **ErrorBoundary in a non-.tsx file (e.g. plain .ts):** still checked.
